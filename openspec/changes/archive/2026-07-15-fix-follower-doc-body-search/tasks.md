## 1. Wire the body channel into the live search path

- [x] 1.1 Pin down which search backend the follow-mode daemon builds (does `initialSearchBackend`'s `graph.SymbolSearcher` branch fire for `store_pg`? does the follower boot use it at all?) and document the finding in the change
- [x] 1.2 Route the postgres backend through `SymbolSearcherBackend` (store-native `SearchSymbols`/`SearchSymbolBundles`, body-aware) ‚Äî or, if follower boot uses a separate path, wire that path equivalently. Root cause was the engine's `Count() > 0` readiness gate (engine.go:504) skipping the already-wired store-routed backend on a follower (Count()==0, never indexes). Added a `search.StoreRouted` marker (implemented by `SymbolSearcherBackend`, forwarded by `Swappable`/`HybridBackend`) so the gate keeps the store-native path live. Verified end-to-end against the live follower: body phrase ‚Üí 8 `kind=doc` hits (was 0).
- [x] 1.3 Benchmark search latency/ranking on a large store before/after the routing change; guard with a fallback flag if regressions appear. The change only alters behavior for a store-routed backend at `Count()==0` ‚Äî i.e. the follower, which was fully broken (name-only substring fallback). The writer's `Count()` is already >0 so its path is byte-for-byte unchanged; no cross-cutting ranking/latency shift. Live follower latencies after the fix: identifier 207ms, body phrase 171ms, NL (521 hits across all repos) 837ms ‚Äî all healthy. Heading matches still surface and rank in for `corpus=docs`. No fallback flag added: the store-native path is strictly more capable than the name-substring fallback for a store-routed backend, and `gatherBackendCandidates` retains its own exact-name/substring tiers when the store returns nothing, so an empty store degrades gracefully without it.

## 2. Preserve identifier tokens in section_text

- [x] 2.1 Locate the normalization that strips underscores from prose-section bodies and preserve identifier-significant characters in the STORED `section_text`. Root cause: `stripMarkdownInline` (internal/parser/languages/markdown_prose.go) used `strings.NewReplacer(‚Ä¶ "_", "" ‚Ä¶)` which blanket-removed every underscore (`branch_track`‚Üí`branchtrack`).
